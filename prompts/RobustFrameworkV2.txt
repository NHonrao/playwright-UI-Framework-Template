Create a robust Playwright test automation framework using the following natural language requirements.
This is an improved version of the original framework prompt, updated based on everything we learned
while building and refining the framework. Please follow these requirements exactly as described.


## Setup Requirements

Please set up a complete test environment with:
1. Node.js and npm for running JavaScript and TypeScript
2. Playwright for browser automation
    - Install Playwright
    - Install browser engines (Chromium, Firefox, WebKit)
    - Set up Playwright test runner
3. TypeScript for better code reliability
4. VS Code extensions for:
    - Playwright Test Explorer
    - Test Running and Debugging
    - Code Intelligence
5. Reporting tools for:
    - HTML test reports
    - Allure detailed reports
    - Screenshot and video capture on failure
    - Test execution logs written to files


## Framework Overview

Please build an end-to-end test automation framework that should:
1. Handle test failures gracefully with screenshots and detailed logs
2. Use clean and simple element finding strategies that are easy to update
3. Support running tests in different browsers
4. Follow good coding practices so tests are easy to read and maintain
5. Include clear error messages that tell you exactly what went wrong and where
6. Keep all test logic inside page classes so test files stay clean and simple
7. Make it easy to add new pages and new test files without repeating yourself


## How to Organise the Project

The project should be organised into clear folders so everything is easy to find:
1. A folder for page classes - one file per page of the website
2. A folder for test files - one file per feature or area being tested
3. A folder for utility helpers such as logging, screenshots, database access and test data
4. A folder for configuration so all settings are in one place
5. A folder for test data files so test data is separate from test logic
6. A folder for shared test setup and teardown logic that is reused across all test files
7. A reports folder where all screenshots, logs, HTML reports and Allure results are saved


## Page Object Model and OOP Rules

The framework must follow the Page Object Model pattern and good coding principles:

1. Every page of the website should have its own class file
    - All the ways to find elements on that page must be inside the page class only
    - Tests should never contain selectors or page element details
    - Selectors should be found by inspecting the real website and confirmed to work

2. Each page class should have two kinds of methods
    - Action methods for doing things on the page such as logging in, clicking buttons or filling forms
    - Verification methods for checking that things are correct on the page such as checking a heading
      is visible or confirming the user is logged in

3. All low-level browser interactions such as clicking, typing, waiting and finding elements should
   live in a shared base class that all page classes inherit from
    - Page classes should call helpers from the base class, not write their own browser code
    - Tests should only call methods on page classes, never interact with the browser directly

4. Test files should be clean and simple
    - A test should only say what to do in plain steps, not how to do it
    - No browser code, no selectors and no waiting logic should appear inside a test file

5. When writing check methods on page classes, always use the real confirmed selector for that page
    - Do not write long lists of guessed selectors hoping one will work
    - Inspect the actual live page first, confirm the selector works, then write it into the class


## Test Writing Format

Every test should follow the Arrange, Act, Assert pattern:
1. Arrange - set up any test data or starting conditions needed
2. Act - perform the actions on the page such as navigating and clicking
3. Assert - check that the expected results are visible on the page

Each action and each check inside a test should be wrapped in a named step so the report
shows exactly what the test was doing at every point.

Each test should also include report labels describing:
- Which product area it belongs to
- Which feature it is testing
- What scenario it covers
- How critical it is
- Which tags apply such as smoke, regression, negative or security


## Shared Test Setup and Teardown

All the common setup and teardown steps that every test needs should be written once in a
shared file and applied to a test suite with a single line of code. This means:

1. There should be no repeated beforeEach or afterEach blocks inside individual test files
2. The shared setup should log the suite name, environment name and base URL at the start
3. Before each test, the shared setup should attach report labels and log which test is starting
4. After each test, the shared setup should:
    - Log whether the test passed or failed
    - If the test failed, take a full page screenshot and save it to the reports folder
    - If the test failed, also attach the screenshot and the current page URL directly to the
      Allure report so it is visible without downloading any files
5. At the end of the suite, the shared setup should log that the suite has finished

When writing a new test suite, only one line is needed to get all of this behaviour.


## Reusable Test Fixtures

The framework should provide ready-made building blocks for tests:
1. A page object fixture for each page class so tests can use them without creating them manually
2. An authenticated session fixture that handles login before the test and logout after the test
    - This fixture should confirm the home page is visible after login before handing control to the test
    - After the test it should attempt to log out and capture a screenshot if the test failed
3. A database helper fixture that connects to the database and disconnects after the test automatically
4. A test data fixture that loads data from files and clears its memory cache after the test
5. A screenshot helper fixture for taking screenshots at specific points during a test

All test files should import these fixtures from a single shared location, not from Playwright directly.


## Test Data Management

Test data should be kept in separate JSON files, not written inside test files:
1. User credentials should be in a users file with different user types such as admin and readonly
2. Test-specific data like invalid credentials or search terms should be in a test data file
3. The framework should load these files automatically and provide them to tests through a helper
4. The helper should remember files it has already loaded so it does not read the same file twice
5. Real credentials should only exist in environment files and never in the test data JSON files


## Logging

The framework should write clear logs so you can see exactly what each test is doing:
1. Every action should be logged before it happens and confirmed with a tick when it succeeds
2. Warnings should be logged when something is skipped or has a non-critical issue
3. Errors should be logged with full details when something fails
4. All logs should be saved to files in the reports folder so you can review them after a run
5. Log output should also appear in the terminal while tests are running


## How to Handle Problems

The framework should be resilient and handle problems gracefully:
1. If a test fails, always take a screenshot and attach it to the report
2. Log a clear message saying what failed and where
3. If a page is slow, wait for the specific key element to appear rather than waiting a fixed time
4. If login is rejected, confirm it by checking the browser stayed on the login page
    - Do not depend only on an error message appearing since different applications show errors differently
5. Actions like logout that run during cleanup should never stop the cleanup from finishing even if they fail
6. Checking for optional elements like error messages should use a short wait time so the test does not
   sit waiting for a long time for something that may not be there
7. Playwright should retry failed tests once when running locally and twice when running in the pipeline,
   and should record a trace and video on the first retry to help with debugging


## How to Describe Page Elements

When I give you a new page to add, I will describe elements in multiple ways to help you
find the right selector by inspecting the live page:

1. By its visible text
    - The exact text shown on the button or link
    - Any nearby text that helps identify it
    - The label or placeholder text

2. By its purpose
    - What the element does such as submitting a form or searching
    - What kind of element it is such as a button, text box or dropdown
    - Where you would expect to find it on the page

3. By its appearance
    - Its colour, size or icon
    - Where it appears on the page such as top, bottom or next to another element
    - Any special state such as disabled or highlighted

4. By its context
    - What section of the page it is in
    - What comes before or after it
    - What it is related to

After I describe an element, you should inspect the real live page to confirm the selector works
before writing it into the page class. Never guess - always verify against the actual page.


## Special Cases to Consider

1. Login and security
    - How to log in and log out
    - When login fails, confirm it by checking the browser is still on the login page
    - How to handle pages that load slowly
    - Do not hardcode credentials anywhere in the code or test data files

2. Forms and Inputs
    - How to fill in forms and clear fields before typing
    - What to do with dropdowns and date pickers
    - What to do if saving fails

3. Messages and Popups
    - What success messages to look for after an action
    - How to handle error messages
    - What to do with confirmation popups and unexpected alerts


## Allure Reporting

Allure reporting should be handled at the framework level, not inside individual tests:
1. The report should show a clear hierarchy grouping tests by product, feature and scenario
2. Every test step should appear in the report with its name and result
3. Failed tests should have the failure screenshot attached directly inside the report
4. The report should show which environment and which browser each test ran on
5. After the pipeline runs the Allure report should be published as an artifact in Azure DevOps
6. The report should also show which attempt number a test was on so retries are clearly visible


## Multiple Environment Handling

The framework should support running tests in different environments with minimal changes:
1. All environment-specific settings such as URLs, credentials and database details should be in
   separate environment files, one file per environment
2. The user should be able to switch environments by setting one variable before running
    - For example setting TEST_ENV to tst or uat should switch everything automatically
3. Running tests locally should work the same way as running them in the pipeline
4. In the pipeline, secret values such as passwords and database credentials should come from the
   Azure DevOps pipeline variable library and never be committed to the repository
5. All configuration should flow through a single config file
    - No test or page class should read environment settings directly


## Project Configuration Needs

1. Package dependencies
    - Use Playwright as the test runner
    - Use TypeScript for all code
    - Use Winston for structured logging
    - Use Allure for detailed reporting
    - Use dotenv for loading environment files
    - Use mssql for Microsoft SQL Server database access
    - Use ESLint and Prettier for code quality and consistent formatting

2. Configuration files needed
    - playwright.config.ts with settings for all three browsers, reporters and timeouts
    - tsconfig.json for TypeScript
    - ESLint config for code style rules
    - Prettier config for consistent formatting
    - Environment files for each environment with all settings in them

3. Project structure
    - Test files go in a tests folder organised by feature area
    - Page classes go in a pages folder
    - Utility helpers go in a utils folder
    - Shared fixture and hook files go in a fixtures folder
    - Test data JSON files go in a data folder
    - All reports, screenshots, videos and logs are saved inside a reports folder

4. Additional tooling
    - Code should be formatted consistently and lint errors should be zero
    - Utility classes should be created for logging, screenshots, database access, test data and waits
    - Microsoft SQL Server connection is needed so tests can run queries to get data from the database
    - Test data should be managed through a separate helper class, never hardcoded in tests

5. Allure Reporting
    - Allure reporting should be configured in playwright.config.ts so it runs automatically
    - After a pipeline run the Allure report should be generated and published as an artifact
    - The report should show full step logs, screenshots and environment details for every test

6. Azure DevOps Pipeline
    - There should be a pipeline YAML file that installs dependencies, installs browsers, writes
      the environment file from secret pipeline variables, runs the tests, and publishes all reports
    - The pipeline should publish JUnit results to the native ADO test dashboard as well as the
      Allure report, Playwright HTML report, screenshots and logs as separate pipeline artifacts
    - All publishing steps should run even if tests fail


## What Good Test Results Look Like

Tests should:
1. Run cleanly and pass consistently when the application is working correctly
2. Tell you exactly what went wrong if they fail, with a screenshot and a clear message
3. Work the same way in Chrome, Firefox and Safari
4. Be easy to update when the website changes because selectors only live in page classes
5. Handle slow-loading pages without timing out unexpectedly
6. Show a clear pass or fail result for every test in both the terminal and the report
7. Leave the browser in a clean state after each test by logging out when needed

## Known Pitfalls — Read These Carefully

These are mistakes that are easy to make and will cause the framework to fail silently or
produce confusing errors. Follow these rules exactly:

1. Playwright beforeEach hook signature
    - When registering a beforeEach hook, the first parameter must always use object destructuring
      even if you do not need any fixtures, write it as ({}, testInfo) not as (_fixtures, testInfo)
    - Playwright requires this specific syntax and will throw an error if you use a plain variable name

2. Allure tag labels
    - allure.tag() only accepts a single string, not an array
    - If you want to add multiple tags, call allure.tag() once for each tag on a separate line
    - Passing an array will cause a TypeScript error

3. Checking for optional elements with a short timeout
    - When checking for something that may or may not be on the page, such as an error message,
      you must write the wait and the text reading directly inside a try/catch block
    - Do not pass the element through a shared helper method that uses the full default timeout
    - If you do, the timeout error will escape the try/catch and the test will hang and fail after
      the full wait period instead of moving on gracefully after a short wait
    - Use a direct waitFor with a timeout of around 3 seconds inside the try/catch instead

4. Verifying that a login was rejected
    - The most reliable way to confirm a failed login is to check that the browser URL still
      contains the login path after the form is submitted
    - Do not make the test depend only on an error message element appearing on screen, because
      different applications show errors in different ways and the element may not exist
    - If an error message does appear, log its text as useful information but do not assert on it

5. Imports in test files
    - Every test file must import test and expect from the custom fixtures file, not from Playwright directly
    - Importing from Playwright directly will bypass all the custom fixtures and the test will not
      have access to the page objects, fixtures or the authenticated session

6. Selectors must come from real DOM inspection
    - Never write a list of guessed selectors hoping one will work
    - Before writing a selector into a page class, open the browser, inspect the live page and
      confirm the exact CSS selector that uniquely identifies the element
    - One confirmed selector per element is better than ten guesses


## How to Write Your Test Request

When asking for a new test, tell me:
1. The website or page you want to test
2. What you want to accomplish written in plain English steps
3. Any login or data setup needed before the test starts
4. What should be visible on the page to confirm the test passed
5. Any negative scenarios to check such as invalid input or access being denied
6. What environment the test should run in

This will help create reliable, easy-to-maintain tests that catch real problems.
